## Project Overview

Green-Cycle simulates a **smart city waste assistant**. A citizen describes a waste item in plain text and the system returns:

1. A **waste category** вЂ” `Recyclable`, `Compost`, or `Hazardous`
2. A **confidence score** вЂ” how certain the model is
3. A **human-readable disposal plan** вЂ” generated by an LLM following simulated city policy

The project demonstrates a complete production-style AI engineering pipeline:

- **ML Engineering** вЂ” NLP preprocessing with spaCy lemmatisation, TF-IDF feature extraction, Logistic Regression with stratified cross-validation and overfitting analysis
- **AI Agent Design** вЂ” logic-based policy decision before LLM invocation, 3-shot prompting (one example per class), safety validation, and a layered fallback chain
- **REST API** вЂ” FastAPI with Pydantic validation, dependency injection via `lru_cache`, and correct HTTP status codes
- **Docker** вЂ” multi-stage build using `python:3.10-slim`, model trained at build time, non-root user, built-in health check

## Docker Deployment

The Dockerfile uses a **multi-stage build**:

- **Builder stage** вЂ” installs all dependencies, copies code and data, runs `python -m app.ml.train` to embed the trained model
- **Runtime stage** вЂ” copies only installed packages and trained artefacts into a fresh slim image, runs as non-root `appuser`

### Build the Image

```bash
docker build -t green-cycle .
```

### Run the Container

```bash
docker run -p 8000:8000 \
  -e LLM_API_KEY=your_groq_api_key_here \
  -e LLM_MODEL=llama-3.1-8b-instant \
  -e LLM_BASE_URL=https://api.groq.com/openai/v1/chat/completions \
  green-cycle
```

API available at `http://localhost:8000`

**Dockerfile design highlights:**

| Feature | Detail |
|---------|--------|
| Base image | `python:3.10-slim` вЂ” minimal footprint |
| Multi-stage | Builder installs + trains; runtime contains only what is needed |
| Layer cleanup | `rm -rf /var/lib/apt/lists/*` after apt install |
| Pip cache | `ENV PIP_NO_CACHE_DIR=1` вЂ” no pip cache in image layers |
| Non-root user | `appuser` created вЂ” server does not run as root |
| Health check | `HEALTHCHECK` polls `/health` every 30s with 3 retries |

## ML Engineering Deep Dive

### Dataset

`data/waste_data.csv` вЂ” 301 balanced samples:

| Class | Count | Percentage |
|-------|-------|------------|
| Recyclable | 101 | 33.6% |
| Compost | 100 | 33.2% |
| Hazardous | 100 | 33.2% |

Synthetic data was used because no publicly labelled waste description dataset exists at this granularity. Items cover vocabulary variation вЂ” for example, `Hazardous` includes "used batteries", "car battery", "old laptop battery", and "lithium battery pack" вЂ” so the model learns the underlying concept, not specific phrases. Balanced classes prevent majority-class bias.

---

### Text Preprocessing

`clean_text()` in `preprocessor.py` is passed as the `preprocessor` argument to `TfidfVectorizer`. This ensures **identical cleaning at training and inference time**, preventing train/serve skew.

| Step | Example | Why |
|------|---------|-----|
| Lowercase | `"Paint Thinner CAN"` в†’ `"paint thinner can"` | "Paint" and "paint" are the same feature |
| Remove non-alpha | `"bottle!!!"` в†’ `"bottle"` | Eliminates punctuation and numeric noise |
| Collapse whitespace | `"old  can"` в†’ `"old can"` | Normalises spacing artefacts |
| spaCy lemmatisation | `"batteries"` в†’ `"battery"` | Maps inflected forms to one base вЂ” shrinks vocabulary |
| Stopword removal | removes "the", "is", "a" | Appear uniformly across all classes вЂ” zero signal |
| Short token filter | drops tokens with len <= 2 | Eliminates noise tokens like "ok", "xx" |

**Why lemmatisation over stemming?**

Stemming applies crude suffix rules: `"batteries"` в†’ `"batteri"` вЂ” not a real word. Lemmatisation uses spaCy's vocabulary and morphological analysis: `"batteries"` в†’ `"battery"`. In a domain where item names carry the primary signal, real word forms are more reliable and interpretable than mangled stems.

**Graceful fallback:** If `en_core_web_sm` is not found, `preprocessor.py` falls back to `spacy.blank("en")` and logs a warning вЂ” the API continues running without crashing.

---

### Feature Extraction вЂ” TF-IDF

| Parameter | Value | Reason |
|-----------|-------|--------|
| `preprocessor` | `clean_text` | NLP cleaning inside vectorizer вЂ” one source of truth for train and inference |
| `ngram_range` | `(1, 1)` | Unigrams only вЂ” bigrams on 301 samples add sparsity without benefit |
| `max_df` | `0.90` | Drops terms in >90% of docs вЂ” too common to distinguish classes |
| `min_df` | `2` | Drops terms in only 1 doc вЂ” likely typos or unique noise |
| `sublinear_tf` | `True` | log(1 + tf) вЂ” reduces outsized influence of very frequent terms |

**Why TF-IDF over CountVectorizer?** CountVectorizer gives raw term counts вЂ” terms frequent everywhere dominate even after stopword removal. TF-IDF down-weights common terms and up-weights class-distinctive ones. "Container" appears in all three categories вЂ” TF-IDF reduces its weight automatically, giving more influence to "battery", "peel", "recycle".

---

### Model Selection вЂ” Logistic Regression vs KNN

| Criterion | Logistic Regression | KNN |
|-----------|---------------------|-----|
| Sparse high-dimensional vectors | Purpose-built вЂ” linear in feature space | Curse of dimensionality вЂ” distances become unreliable |
| Overfitting control | L2 regularisation via C parameter | None вЂ” memorises training data |
| Probability output | Clean predict_proba for confidence | Noisy vote-based estimates |
| Inference speed | O(1) вЂ” single dot product | O(n) вЂ” computes distance to all training samples |

TF-IDF vectors are 200вЂ“400 dimensions for this dataset. In high-dimensional space, Euclidean distance (KNN's foundation) converges вЂ” all pairwise distances become similar, making neighbours meaningless. Logistic Regression performs linear classification in that space, which is precisely its strength.

**Parameters used:**

| Parameter | Value | Reason |
|-----------|-------|--------|
| `C` | `1.0` | Default вЂ” confirmed optimal by C-value comparison in notebook |
| `class_weight` | `"balanced"` | Adjusts penalties for minor class imbalance |
| `solver` | `"lbfgs"` | Best for multinomial problems |
| `max_iter` | `1000` | Ensures convergence on full vocabulary |
| `random_state` | `42` | Reproducible results |

## Generalisation Evidence and Evaluation

### Train vs Test Accuracy

| Metric | Value |
|--------|-------|
| Training samples | 240 (80% stratified) |
| Test samples | 61 (20% stratified) |
| Train Accuracy | ~0.9542 |
| Test Accuracy | ~0.9180 |
| Gap | ~0.0362 |

A gap of ~3.6% is within the healthy threshold of 5%. The model learns genuine patterns and applies them to unseen examples. A gap above 10% would signal overfitting (memorising training data). Stratified split (`stratify=y`) ensures the same class proportions in both sets вЂ” no class is over- or under-represented in evaluation.

---

### 5-Fold Stratified Cross-Validation

| Metric | Value |
|--------|-------|
| CV Mean Accuracy | ~0.9136 |
| CV Std | ~0.0201 |
| Min Fold | ~0.900 |
| Max Fold | ~0.933 |

A single 80/20 split could give misleadingly optimistic results if the held-out set happens to be easy. Cross-validation rotates the test set five times вЂ” every sample is tested exactly once вЂ” giving a more honest generalisation estimate.

- CV mean ~0.914 close to test ~0.918 в†’ the test result was not a lucky split
- CV std ~0.020, below the 0.05 healthy threshold в†’ model is stable across splits
- Consistent fold range 0.90вЂ“0.93 в†’ no single fold is drastically different

---

|-------|-----------|--------|----------|---------|

### Learning Curve

Plotted in `green_cycle_ml.ipynb` вЂ” train and validation accuracy vs training set size:

- Curves converge at full dataset size вЂ” no growing gap, rules out overfitting
- Both plateau above 0.90 вЂ” sufficient signal in data, rules out underfitting

---

### Confidence Threshold

Predictions below **0.55** confidence are returned as `"Uncertain"`. Applied in `WasteAuditService` вЂ” single source of truth. With three classes, random prediction produces ~0.33 confidence. A 0.55 threshold ensures meaningful signal above random before committing to a label вЂ” prevents overconfident wrong predictions on out-of-vocabulary inputs and skips the LLM call entirely when uncertain.

## AI Agent Design

### Logic-Based Decision Sequence (Agency)

The agent makes rule-based decisions before calling the LLM:

```
1.  Sanitise input
2.  Detect meaningless input (len < 3 or <= 2 unique chars)
3.  Classify via ML model
4.  Gate on confidence threshold (< 0.55 в†’ Uncertain, stop here)
5.  Look up category-specific city policy (dictionary lookup)
6.  Append safety note per category:
      Hazardous  в†’ "Ensure it is kept away from children and pets."
      Recyclable в†’ "Make sure it is clean and dry before recycling."
7.  Build 3-shot prompt
8.  Call LLM API
9.  Validate response (length, forbidden phrases, category guardrail)
10. Return LLM response or policy fallback
```

### City Policy Simulation

Defined in `app/config.py` as `CITY_POLICIES` dict, retrieved by `CityPolicyService.get_policy(category)`. This is the deliberate logic-based decision step before the LLM. The pattern can be replaced with a real municipal API without touching any other component.

| Category | Policy |
|----------|--------|
| Recyclable | Rinse the item and place it in the blue recycling bin. |
| Compost | Place the item in the green compost bin. |
| Hazardous | Seal in a leak-proof container and take to the Hazardous Waste Facility. |

---

### 3-Shot Prompt Structure

```
You are a waste disposal assistant.

Your task is to provide a safe and correct disposal plan based on the waste
description and category. Follow the provided city policy when generating the plan.

Examples:

Description: empty plastic bottle
Category: Recyclable
City Policy: Plastic bottles must be rinsed and placed in the blue bin.
Disposal Plan: Rinse the bottle and put it in the blue recycling bin.

Description: banana peel
Category: Compost
City Policy: Organic waste should go to compost bins.
Disposal Plan: Place the peel in the compost bin.

Description: used batteries
Category: Hazardous
City Policy: Batteries must be taken to a hazardous waste drop off.
Disposal Plan: Take the batteries to the hazardous waste collection site.

Now generate the disposal plan.

Description: {description}
Category: {category}
City Policy: {policy}

Disposal Plan:
```

Three examples вЂ” one per class вЂ” demonstrate the expected format and length to the LLM.
