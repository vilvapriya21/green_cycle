## Project Overview

Green-Cycle simulates a **smart city waste assistant**. A citizen describes a waste item in plain text and the system returns:

1. A **waste category** — `Recyclable`, `Compost`, or `Hazardous`
2. A **confidence score** — how certain the model is
3. A **human-readable disposal plan** — generated by an LLM using simulated city policy

The project demonstrates a complete production-style AI engineering pipeline:

- **ML Engineering** — NLP preprocessing, TF-IDF feature extraction, Logistic Regression with cross-validation and overfitting analysis
- **AI Agent Design** — logic-based policy decision, few-shot LLM prompting, safety validation, and layered fallback handling
- **REST API** — FastAPI with Pydantic validation, dependency injection, and correct HTTP status codes
- **Docker Containerisation** — optimised slim image with non-root user and runtime secret injection

## System Workflow

```
User Text Input
      │
      ▼
FastAPI Endpoint  (/classify  or  /disposal)
      │
      ▼
WasteAuditService  (orchestration layer)
      │
      ├── Input sanitisation + meaningless input detection
      │
      ▼
NLP Preprocessing  (preprocessor.py)
  - Lowercase
  - Remove non-alpha characters
  - spaCy lemmatisation
  - Stopword and punctuation removal
      │
      ▼
TF-IDF Vectorisation
      │
      ▼
Logistic Regression Classifier
      │
      ▼
Predicted Category + Confidence Score
      │
      ├── Confidence < 0.55  →  return "Uncertain"  (skip LLM)
      │
      ▼
CityPolicyService  (logic-based category → policy lookup)
      + category-specific safety note appended
      │
      ▼
PromptBuilder  (few-shot prompt with policy injected)
      │
      ▼
LLM API Call  (Groq / OpenAI-compatible)
      │
      ├── API key missing       →  fallback to policy text
      ├── Timeout / network err →  fallback to policy text
      ├── HTTP 429 rate limit   →  fallback to policy text
      ├── Response too short    →  fallback to policy text
      └── Unsafe phrase found   →  fallback to policy text
      │
      ▼
JSON Response  { text, category, confidence, disposal_plan }
```

**Dockerfile design decisions:**

- Runs `rm -rf /var/lib/apt/lists/*` after apt install to reduce layer size

## ML Engineering Deep Dive

### Dataset

The training dataset (`data/waste_data.csv`) contains **301 synthetic samples**:

| Class | Count | Percentage |
|-------|-------|------------|
| Recyclable | 101 | 33.6% |
| Compost | 100 | 33.2% |
| Hazardous | 100 | 33.2% |

Balanced classes prevent the model from developing majority-class bias. Synthetic data was used because no publicly labelled waste description dataset exists at this granularity. Items were designed to cover vocabulary variation — for example, Hazardous includes "used batteries", "car battery", "old laptop battery", and "lithium battery pack" — so the model learns the concept rather than memorising specific phrases.

---

### Text Preprocessing — Why Each Step Matters

The `clean_text()` function in `preprocessor.py` is passed directly into `TfidfVectorizer` as its `preprocessor` argument. This guarantees **identical cleaning at training time and inference time**, eliminating train/serve skew — a common production failure.

| Step | Example | Why it matters |
|------|---------|----------------|
| Lowercase | `"Paint Thinner CAN"` → `"paint thinner can"` | Prevents treating "Paint" and "paint" as different features |
| Remove non-alpha | `"bottle!!!"` → `"bottle"` | Keeps only meaningful word content |
| spaCy lemmatisation | `"batteries"` → `"battery"` | Maps inflected forms to one token — reduces vocabulary size |
| Stopword removal | removes "the", "is", "a" | These appear in all classes — zero discriminating power |
| Short token filter | drops tokens with length ≤ 2 | Removes noise like "ok", "xx" |

**Why lemmatisation over stemming?**

Stemming applies crude suffix rules: `"batteries"` → `"batteri"` — not a real word. Lemmatisation uses vocabulary lookup and morphological analysis to produce real base forms: `"batteries"` → `"battery"`. This produces cleaner, interpretable features. In a domain where specific item names matter (e.g., "battery" vs "chemical"), real word forms are more reliable than mangled stems.

---

### Feature Extraction — TF-IDF

**Why TF-IDF over CountVectorizer?**

CountVectorizer gives raw counts, so high-frequency words dominate even after stopword removal. TF-IDF down-weights terms appearing across many documents and up-weights class-distinctive terms. For example, "container" appears in all three classes — TF-IDF reduces its weight automatically, letting more specific terms like "hazardous", "battery", and "peel" carry more signal.

**Key vectorizer parameters:**

| Parameter | Value | Reason |
|-----------|-------|--------|
| `preprocessor` | `clean_text` | Applies NLP cleaning inside the vectorizer — single point of truth for both train and inference |
| `ngram_range` | `(1, 1)` | Unigrams only — bigrams on 301 samples add sparsity without meaningful benefit |
| `max_df` | `0.90` | Drops terms in >90% of documents — too common to be discriminating |
| `min_df` | `2` | Drops terms in only 1 document — likely typos or noise |
| `sublinear_tf` | `True` | Log-normalises term frequency — reduces outlier effect of very high counts |

---

### Model Selection — Logistic Regression vs KNN

| Criterion | Logistic Regression | KNN |
|-----------|---------------------|-----|
| High-dimensional sparse TF-IDF vectors | Handles natively — linear model in feature space | Curse of dimensionality — distances become unreliable |
| Overfitting control | Built-in L2 regularisation via `C` | No regularisation — memorises training data easily |
| Probability output | Yes — clean class probabilities for confidence scoring | Noisy vote-based estimates |
| Inference speed | O(1) — single dot product | O(n) — computes distance to all training samples |
| Interpretability | Coefficients show which words drive each class | Opaque |

TF-IDF vectors are typically 200–800 dimensions for a dataset this size. In high-dimensional space, all points tend toward equal Euclidean distance — KNN neighbours become meaningless. Logistic Regression treats the problem as linear classification in that space, which is precisely its strength.

**Why C=1.0?**

The `C` parameter is the inverse of regularisation strength — lower C means stronger regularisation and smaller coefficients. C=1.0 (scikit-learn default) works well here because preprocessing already constrains the vocabulary (stopwords removed, min_df=2 eliminates rare noise). The C-value comparison in `green_cycle_ml.ipynb` confirms C=1.0 achieves the best balance of train and test accuracy for this dataset.

## Generalisation Evidence and Evaluation

### Train vs Test Accuracy

| Metric | Value |
|--------|-------|
| Train Accuracy | ~0.9542 |
| Test Accuracy | ~0.9180 |
| Gap | ~0.0362 |

**What the gap tells us:** A train-test gap of ~3.6% is within the healthy threshold of 5%. The model learns genuine patterns from training data and applies them successfully to held-out examples. A gap above 10% would indicate overfitting (memorising training data). A gap near zero on a small dataset would be suspicious — it could mean the test set is too similar to training data, or that the model is too simple.

---

### 5-Fold Stratified Cross-Validation

| Fold | Accuracy |
|------|----------|
| Fold 1 | ~0.91 |
| Fold 2 | ~0.93 |
| Fold 3 | ~0.90 |
| Fold 4 | ~0.92 |
| Fold 5 | ~0.91 |
| **Mean** | **~0.914** |
| **Std** | **~0.020** |

**Why cross-validation matters beyond a single split:**

A single 80/20 split could give misleadingly optimistic results if the test set happens to contain easy examples. Stratified cross-validation rotates the held-out set across all 5 folds, so every sample is used for testing exactly once. The result is a more honest estimate of real-world performance.

**What these numbers prove:**

A CV mean of ~0.914, close to the test accuracy of ~0.918, confirms the test split result was not a lucky outcome — the model consistently achieves this performance across different data splits. A standard deviation of ~0.020 is below the healthy threshold of 0.05, confirming the model is stable and not sensitive to which particular examples end up in the test set.

---

### Learning Curve Interpretation

The learning curve (in `green_cycle_ml.ipynb`) plots train and validation accuracy as training set size increases.

- Train and validation accuracy **converge** as data increases — no growing gap at full dataset size, ruling out overfitting
- Both curves plateau above 0.90 — sufficient signal exists in the data, ruling out underfitting
- Both curves still show slight upward trend at 301 samples — performance would improve further with more data

### Per-Class Classification Report

| Class | Precision | Recall | F1-Score |
|-------|-----------|--------|----------|
| Recyclable | ~0.93 | ~0.93 | ~0.93 |
| Compost | ~0.91 | ~0.90 | ~0.91 |
| Hazardous | ~0.92 | ~0.93 | ~0.92 |

All three classes perform consistently — no class is being systematically ignored or misclassified. This is partly ensured by `class_weight="balanced"` in Logistic Regression, which adjusts the penalty for misclassifying minority classes automatically.

---

### Confidence Threshold

Predictions with model confidence below **0.55** are returned as `"Uncertain"` rather than a potentially wrong category. This is applied in `WasteAuditService` as the single source of truth — the classifier itself does not apply gating.

**Why 0.55?** With three classes, a random prediction would produce ~0.33 confidence. A threshold of 0.55 ensures the model has meaningful signal above random before committing to a label. This prevents overconfident wrong predictions on completely unknown inputs such as nonsense text or items far outside the training vocabulary.

## AI Agent Design

### Logic-Based Decisions Before LLM

The agent follows a deliberate decision sequence before the LLM is called:

```
1. Sanitise and validate input text
2. Classify text via ML model
3. Gate on confidence threshold (< 0.55 → return Uncertain, skip LLM)
4. Look up category-specific city policy from dictionary
5. Append category-specific safety note:
       Hazardous  → "Keep away from children and pets."
       Recyclable → "Ensure item is clean and dry before recycling."
6. Build few-shot prompt with description, category, and policy
7. Call LLM API
8. Validate LLM response:
       - Length check (< 20 chars → fallback)
       - Forbidden phrase check → fallback
       - Category guardrail (Hazardous told to "recycle" → fallback)
9. Return LLM response or policy fallback
```

This multi-step logic means the LLM is never called blindly. The system makes category-specific rule-based decisions first — this is the "agency" component of the agent architecture.

### Few-Shot Prompt Structure

```
You are a waste disposal assistant.

Your task is to provide a safe and correct disposal plan based on the waste
description and category. Follow the provided city policy when generating the plan.

Examples:

Description: empty plastic bottle
Category: Recyclable
City Policy: Plastic bottles must be rinsed and placed in the blue bin.
Disposal Plan: Rinse the bottle and put it in the blue recycling bin.

Description: used batteries
Category: Hazardous
City Policy: Batteries must be taken to a hazardous waste drop off.
Disposal Plan: Take the batteries to the hazardous waste collection site.

Now generate the disposal plan.

Description: {description}
Category: {category}
City Policy: {policy}

Disposal Plan:
```

The two worked examples demonstrate the expected output format, length, and tone — guiding the LLM to produce concise actionable instructions rather than lengthy explanations.

### Fallback Chain

| Failure Scenario | System Response |
|------------------|-----------------|
| `LLM_API_KEY` not set | Skip LLM, return policy text |
| HTTP 429 rate limit | Log warning, return policy text |
| HTTP non-200 response | Log error with status and body, return policy text |
| Request timeout | Log warning, return policy text |
| Network error | Log exception, return policy text |
| Response shorter than 20 characters | Return policy text |
| Unsafe phrase detected in response | Log warning, return policy text |
| Hazardous item advised to recycle | Override to policy text |

## Security

- **No hardcoded secrets** — all API keys and configuration are loaded from environment variables via `pydantic-settings`
- **Docker runtime injection** — keys passed as `-e` flags at `docker run`, never stored in the image
- **`.gitignore`** — `.env` is excluded from version control
- **Non-root container user** — Dockerfile creates `appuser` and switches to it before starting the server
- **Input length cap** — `WasteAuditService` truncates inputs over 500 characters before processing
- **Pydantic validation** — `WasteRequest` enforces `min_length=1` and `max_length=500` at the API boundary before any processing begins

## Possible Improvements

### ML

| Improvement | Expected Impact |
|-------------|-----------------|
| Expand to 1000+ real-world labelled samples | Better generalisation on edge cases and unusual descriptions |
| Add bigrams `ngram_range=(1,2)` with larger dataset | Captures two-word phrases like "motor oil" as single features |
| Try LinearSVC or SGDClassifier | Often marginally outperforms Logistic Regression on sparse text |
| Add a fourth "General Waste" class | More realistic coverage — not all items fit three categories |
| Replace TF-IDF with sentence transformers (e.g. `all-MiniLM-L6-v2`) | Semantic embeddings handle synonyms and paraphrases that TF-IDF misses |

### Agent

| Improvement | Expected Impact |
|-------------|-----------------|
| Replace policy dictionary with a real municipal REST API | Production-ready integration with actual city waste rules |
| Add Retrieval-Augmented Generation (RAG) | LLM retrieves specific local regulations rather than generic text |
| Expand few-shot examples to 4-5 per class | More stable, varied LLM output on edge cases |
| Add prompt response caching (Redis) | Reduces LLM API cost for repeated identical inputs |
| Switch to async LLM calls using `httpx` | Better concurrency under high request load |

### System

| Improvement | Expected Impact |
|-------------|-----------------|
| Add Redis result caching | Avoids re-running ML and LLM for repeated queries |
| Kubernetes deployment with horizontal scaling | Production-grade availability and load distribution |
| Image-based classification (computer vision) | Users photograph items instead of typing descriptions |
| CI/CD pipeline via GitHub Actions | Automated test, build, and deploy on every push |
| Prometheus metrics endpoint | Monitor request latency, confidence distributions, fallback rates in production |
