## Project Overview

Green-Cycle is a small example project for classifying waste descriptions and suggesting disposal steps. A citizen describes a waste item in plain text and the system returns:

1. A **waste category** — `Recyclable`, `Compost`, or `Hazardous`
2. A **confidence score** — how certain the model is
3. A **human-readable disposal plan** — generated by an LLM following simulated city policy

This project combines text classification, a simple LLM-based disposal assistant, and a FastAPI service:

- **ML Engineering** — NLP preprocessing with spaCy lemmatisation, TF-IDF feature extraction, Logistic Regression with stratified cross-validation and overfitting analysis
- **AI Agent Design** — Rule-based policy lookup before the LLM call, simple 3-shot prompting (one example per class), response validation, and fallback to policy text when the LLM fails
- **REST API** — FastAPI with Pydantic validation, dependency injection via `lru_cache`, and correct HTTP status codes
- **Docker** — multi-stage build using `python:3.10-slim`, model trained at build time, non-root user, built-in health check

## Processing Workflow

```
User Text Input
      │
      ▼
FastAPI Endpoint  (/classify  or  /disposal)
      │
      ▼
WasteAuditService  (orchestration layer)
      │
      ├── Input sanitisation + meaningless input detection
      │
      ▼
NLP Preprocessing  (clean_text in preprocessor.py)
  - Lowercase
  - Remove non-alphabetical characters
  - Collapse whitespace
  - spaCy lemmatisation
  - Stopword and punctuation removal
  - Short token filtering (length <= 2 dropped)
      │
      ▼
TF-IDF Vectorisation  (fitted on training data)
      │
      ▼
Logistic Regression Classifier  (predict_proba)
      │
      ▼
Predicted Category + Confidence Score
      │
      ├── Confidence < 0.55  →  return "Uncertain"  (LLM skipped)
      │
      ▼
CityPolicyService  (category to policy text lookup)
      + category-specific safety note appended
      │
      ▼
PromptBuilder  (3-shot prompt with policy injected)
      │
      ▼
LLM API Call  (Groq / OpenAI-compatible endpoint)
      │
      ├── API key missing        →  fallback to policy text
      ├── HTTP 429 rate limit    →  fallback to policy text
      ├── HTTP non-200 response  →  fallback to policy text
      ├── Timeout / network err  →  fallback to policy text
      ├── Response < 20 chars    →  fallback to policy text
      ├── Unsafe phrase detected →  fallback to policy text
      └── Hazardous told recycle →  fallback to policy text
      │
      ▼
JSON Response  { text, category, confidence, disposal_plan }
```

## Docker Deployment

The Dockerfile uses a **multi-stage build**:

- **Builder stage** — installs all dependencies, copies code and data, runs `python -m app.ml.train` to embed the trained model
- **Runtime stage** — copies only installed packages and trained artefacts into a fresh slim image, runs as non-root `appuser`

### Build the Image

```bash
docker build -t green-cycle .
```

### Image Details

- **Base image:** `python:3.10-slim`
- **Multi-stage build:** used to reduce the final runtime image
- **Final image size:** ~963 MB

### Run the Container

Option 1 — Using environment variables

```bash
docker run --name green-cycle-api -p 8000:8000 \
  -e LLM_API_KEY=your_groq_api_key_here \
  -e LLM_MODEL=llama-3.1-8b-instant \
  -e LLM_BASE_URL=https://api.groq.com/openai/v1/chat/completions \
  green-cycle
```

Option 2 — Using .env file (recommended)

```bash
docker run --env-file .env -p 8000:8000 green-cycle
```

API available at `http://localhost:8000`

**Dockerfile design highlights:**

| Feature | Detail |
|---------|--------|
| Base image | `python:3.10-slim` — lightweight base image |
| Multi-stage | Builder installs + trains; runtime contains only what is needed |
| Layer cleanup | `rm -rf /var/lib/apt/lists/*` after apt install |
| Pip cache | `ENV PIP_NO_CACHE_DIR=1` — no pip cache in image layers |
| Non-root user | `appuser` created — server does not run as root |
| Health check | `HEALTHCHECK` polls `/health` every 30s with 3 retries |
| Secret security | API keys provided at runtime using `-e` flags or `--env-file` — never stored in the image |

## ML Design and Implementation

### Dataset

`data/waste_data.csv` — 301 balanced samples:

| Class | Count | Percentage |
|-------|-------|------------|
| Recyclable | 101 | 33.6% |
| Compost | 100 | 33.2% |
| Hazardous | 100 | 33.2% |

Synthetic data was used because no publicly labelled waste description dataset exists at this granularity. Items cover vocabulary variation — for example, `Hazardous` includes "used batteries", "car battery", "old laptop battery", and "lithium battery pack" — so the model learns the underlying concept, not specific phrases. Balanced classes prevent majority-class bias.

---

### Text Preprocessing

`clean_text()` in `preprocessor.py` is passed as the `preprocessor` argument to `TfidfVectorizer`. This ensures **identical cleaning at training and inference time**, preventing train/serve skew.

| Step | Example | Why |
|------|---------|-----|
| Lowercase | `"Paint Thinner CAN"` → `"paint thinner can"` | "Paint" and "paint" are the same feature |
| Remove non-alpha | `"bottle!!!"` → `"bottle"` | Eliminates punctuation and numeric noise |
| Collapse whitespace | `"old  can"` → `"old can"` | Normalises spacing artefacts |
| spaCy lemmatisation | `"batteries"` → `"battery"` | Maps inflected forms to one base — shrinks vocabulary |
| Stopword removal | removes "the", "is", "a" | Appear uniformly across all classes — zero signal |
| Short token filter | drops tokens with len <= 2 | Eliminates noise tokens like "ok", "xx" |

**Why lemmatisation over stemming?**

Stemming applies crude suffix rules: `"batteries"` → `"batteri"` — not a real word. Lemmatisation uses spaCy's vocabulary and morphological analysis: `"batteries"` → `"battery"`. In a domain where item names carry the primary signal, real word forms are more reliable and interpretable than mangled stems.

**Graceful fallback:** If `en_core_web_sm` is not found, `preprocessor.py` falls back to `spacy.blank("en")` and logs a warning — the API continues running without crashing.

---

### Feature Extraction — TF-IDF

| Parameter | Value | Reason |
|-----------|-------|--------|
| `preprocessor` | `clean_text` | NLP cleaning inside vectorizer — one source of truth for train and inference |
| `ngram_range` | `(1, 1)` | Unigrams only — bigrams on 301 samples add sparsity without benefit |
| `max_df` | `0.90` | Drops terms in >90% of docs — too common to distinguish classes |
| `min_df` | `2` | Drops terms in only 1 doc — likely typos or unique noise |
| `sublinear_tf` | `True` | log(1 + tf) — reduces outsized influence of very frequent terms |

**Why TF-IDF over CountVectorizer?** CountVectorizer gives raw term counts — terms frequent everywhere dominate even after stopword removal. TF-IDF down-weights common terms and up-weights class-distinctive ones. "Container" appears in all three categories — TF-IDF reduces its weight automatically, giving more influence to "battery", "peel", "recycle".

---

### Model Selection — Logistic Regression vs KNN

| Criterion | Logistic Regression | KNN |
|-----------|---------------------|-----|
| Sparse high-dimensional vectors | Purpose-built — linear in feature space | Curse of dimensionality — distances become unreliable |
| Overfitting control | L2 regularisation via C parameter | None — memorises training data |
| Probability output | Clean predict_proba for confidence | Noisy vote-based estimates |
| Inference speed | O(1) — single dot product | O(n) — computes distance to all training samples |

TF-IDF vectors are 200–400 dimensions for this dataset. In high-dimensional space, Euclidean distance (KNN's foundation) converges — all pairwise distances become similar, making neighbours meaningless. Logistic Regression performs linear classification in that space, which is precisely its strength.

**Parameters used:**

| Parameter | Value | Reason |
|-----------|-------|--------|
| `C` | `1.0` | Default — confirmed optimal by C-value comparison in notebook |
| `class_weight` | `"balanced"` | Adjusts penalties for minor class imbalance |
| `solver` | `"lbfgs"` | Best for multinomial problems |
| `max_iter` | `1000` | Ensures convergence on full vocabulary |
| `random_state` | `42` | Reproducible results |

## Model Evaluation

### Train vs Test Accuracy

| Metric | Value |
|--------|-------|
| Training samples | 240 (80% stratified) |
| Test samples | 61 (20% stratified) |
| Train Accuracy | ~0.9542 |
| Test Accuracy | ~0.9180 |
| Gap | ~0.0362 |

A gap of ~3.6% is within the healthy threshold of 5%. The model learns genuine patterns and applies them to unseen examples. A gap above 10% would signal overfitting (memorising training data). Stratified split (`stratify=y`) ensures the same class proportions in both sets — no class is over- or under-represented in evaluation.

---

### 5-Fold Stratified Cross-Validation

| Metric | Value |
|--------|-------|
| CV Mean Accuracy | ~0.9136 |
| CV Std | ~0.0201 |
| Min Fold | ~0.900 |
| Max Fold | ~0.933 |

A single 80/20 split could give misleadingly optimistic results if the held-out set happens to be easy. Cross-validation rotates the test set five times — every sample is tested exactly once — giving a more honest generalisation estimate.

- CV mean ~0.914 close to test ~0.918 → the test result was not a lucky split
- CV std ~0.020, below the 0.05 healthy threshold → model is stable across splits
- Consistent fold range 0.90–0.93 → no single fold is drastically different

---

### Per-Class Classification Report

| Class | Precision | Recall | F1-Score | Support |
|-------|-----------|--------|----------|---------|
| Compost | ~0.91 | ~0.90 | ~0.91 | 20 |
| Hazardous | ~0.93 | ~0.93 | ~0.93 | 20 |

---

### Learning Curve

Plotted in `green_cycle_ml.ipynb` — train and validation accuracy vs training set size:

- Curves converge at full dataset size — no growing gap, rules out overfitting
- Both plateau above 0.90 — sufficient signal in data, rules out underfitting
- Slight upward trend at 301 samples — more data would improve performance

---

### Confidence Threshold

Predictions below **0.55** confidence are returned as `"Uncertain"`. Applied in `WasteAuditService` — single source of truth. With three classes, random prediction produces ~0.33 confidence. A 0.55 threshold ensures meaningful signal above random before committing to a label — prevents overconfident wrong predictions on out-of-vocabulary inputs and skips the LLM call entirely when uncertain.

## AI Agent Design

### Agent Decision Flow

The agent makes rule-based decisions before calling the LLM:

```
1.  Sanitise input
2.  Detect meaningless input (len < 3 or <= 2 unique chars)
3.  Classify via ML model
4.  Gate on confidence threshold (< 0.55 → Uncertain, stop here)
5.  Look up category-specific city policy (dictionary lookup)
6.  Append safety note per category:
      Hazardous  → "Ensure it is kept away from children and pets."
      Recyclable → "Make sure it is clean and dry before recycling."
7.  Build 3-shot prompt
8.  Call LLM API
9.  Validate response (length, forbidden phrases, category guardrail)
10. Return LLM response or policy fallback
```

### City Policy Simulation

Defined in `app/config.py` as `CITY_POLICIES` dict, retrieved by `CityPolicyService.get_policy(category)`. This is the deliberate logic-based decision step before the LLM. The pattern can be replaced with a real municipal API without touching any other component.

| Category | Policy |
|----------|--------|
| Recyclable | Rinse the item and place it in the blue recycling bin. |
| Compost | Place the item in the green compost bin. |
| Hazardous | Seal in a leak-proof container and take to the Hazardous Waste Facility. |

---

### 3-Shot Prompt Structure

```
You are a waste disposal assistant.

Your task is to provide a safe and correct disposal plan based on the waste
description and category. Follow the provided city policy when generating the plan.

Examples:

Description: empty plastic bottle
Category: Recyclable
City Policy: Plastic bottles must be rinsed and placed in the blue bin.
Disposal Plan: Rinse the bottle and put it in the blue recycling bin.

Description: banana peel
Category: Compost
City Policy: Organic waste should go to compost bins.
Disposal Plan: Place the peel in the compost bin.

Description: used batteries
Category: Hazardous
City Policy: Batteries must be taken to a hazardous waste drop off.
Disposal Plan: Take the batteries to the hazardous waste collection site.

Now generate the disposal plan.

Description: {description}
Category: {category}
City Policy: {policy}

Disposal Plan:
```

Three examples — one per class — demonstrate the expected format and length to the LLM.

### Fallback Chain

| Failure | Handling |
|---------|----------|
| `LLM_API_KEY` not set | Skip LLM, return policy text |
| HTTP 429 rate limit | Log warning, return policy text |
| HTTP non-200 response | Log error + body, return policy text |

| Network exception | Log exception, return policy text |
| Response < 20 characters | Return policy text |
| Forbidden phrase in response | Log warning, return policy text |

## Possible Improvements

### ML

| Improvement | Expected Impact |
|-------------|-----------------|
| Expand to 1000+ real-world labelled samples | Better generalisation on edge cases |
| Add bigrams `ngram_range=(1,2)` with larger dataset | Captures "motor oil", "paint thinner" as single features |
| Try LinearSVC or SGDClassifier | Often marginally outperforms LR on sparse text |
| Add "General Waste" as a fourth class | More realistic coverage |
| Replace TF-IDF with sentence transformers | Semantic embeddings handle synonyms TF-IDF misses |
